# Superpowers：Agent协同框架

https://github.com/obra/superpowers

背景：当前的智能体（CC、Codex、Cursor...）虽然能够快速生成大量代码，但往往缺乏系统性的设计、严密的测试和可维护的架构。

Superpowers就是一个专门针对这些智能体设计的“代理技能框架”（Agentic Skills Framework）与软件开发方法论应运而生。该框架不仅提供了一套可组合的工具集，更通过强制性的工程纪律，将资深软件工程师的职业操守与工作流程注入到 AI 智能体的核心行为逻辑中。

## Superpowers 的工作流概览

传统的 AI 辅助编程通常是碎片化的、开发者提出一个具体的编码请求，AI 给出一段对应的代码片段。这种模式在处理复杂系统时极易失效，因为智能体缺乏对全局目标的持久关注（思考-行动-观察 循环架构），在循环处理一个任务多次调用工具后，此期间产生的大量 context 会让Agent 忘记最初的目标。

Superpower 的工作流：

1. **brainstorming（头脑风暴）** - 在编写代码之前激活。通过提问完善粗略的想法，探索替代方案，分部分展示设计以供验证。保存设计文档。

2. **using-git-worktrees（使用 git worktree）** - 设计批准后激活。在新分支上创建隔离的工作空间，运行项目设置，验证干净的测试基线。

3. **writing-plans（编写计划）** - 设计批准后激活。将工作分解为可管理的任务（每个任务 2-5 分钟）。每个任务都有精确的文件路径、完整的代码和验证步骤。

4. **subagent-driven-development（子代理驱动开发）** 或 **executing-plans（执行计划）** - 有计划时激活。为每个任务分配一个新的子代理，进行两阶段审查（规格合规性，然后是代码质量），或批量执行并设置人工检查点。

5. **test-driven-development（测试驱动开发）** - 在实现过程中激活。强制执行 RED-GREEN-REFACTOR：编写失败的测试，观察它失败，编写最少的代码，观察它通过，提交。删除在测试之前编写的代码。

6. **requesting-code-review（请求代码审查）** - 在任务之间激活。根据计划进行审查，按严重程度报告问题。严重问题会阻止进度。

7. **finishing-a-development-branch（完成开发分支）** - 任务完成时激活。验证测试，提供选项（合并/PR/保留/丢弃），清理 worktree。

这是**强制性的工作流程，而非建议**。可以看到在Superpower 的语境中，任何代码的生成都不是无端产生的，它必须有明确的需求来源（通过**brainstorming**确认）、详细的实施计划（**writing-plans** 确认）、严密的验证手段（**TDD测试驱动开发**）。

## Superpower是如何做到让大模型每次都遵循工作流

**Agent在执行任何任务之前都会检查相关技能 Skill。** 通过一系列高度聚焦的“Skill技能”来增强其特定领域的能力。智能体会根据当前任务的上下文自动加载并遵循这些规则。官方技能库涵盖了四个关键领域：测试（专注于 TDD 循环及反模式识别）、调试（包含四阶段根因分析法）、协作（管理需求细化、计划执行与并行协作）以及元技能（用于自我优化和新技能编写）。

- **防止Agent 表现的像“急躁的初级开发”**：传统 AI Agent 编程助手接到任务后会立即行动生成大量代码，没有进行架构的思考。就导致生成的代码在语法上正确，但是没有复杂系统应有的架构。因此 Superpower 要求智能体在写代码前进行提问，挖掘出隐藏的需求和条件。
- **消除 AI 的沉没成本**：当 AI 生成一段有缺陷的代码后，它往往倾向于在上面不断打补丁，而不是推倒重来。而 Superpower 的 TDD Skill 包含一条准则：如果生产代码在测试通过前就已经存在，则必须删除它。这就会强迫 AI 回到最简单的逻辑，从而产物更清晰的代码。
- **应对长任务的上下文雪崩**：随着对话轮次的增加，会话的上下文 context 会让 LLM 的注意力下降，导致输出内容的质量下降。而 Superpower 会通过将任务分解为 2-5 分钟的小闭环，并将其写进本地文件，让 LLM 的注意力集中在微小的改动上，从而提高 token 的利用率和执行准确率。

## 适用场景

- **复杂新功能的端到端开发**：当需要在一个大型现有项目中增加一个涉及多个文件、多种逻辑交互的新特性时，Superpowers 的“头脑风暴-计划-子智能体执行”流程能确保新功能与旧架构完美融合
- **大规模重构与架构迁移**：例如将一个过时的前端框架升级到最新版本，或者进行数据库模式的重大调整。这类任务风险极高，Superpowers 详尽的迁移路线图和每一步的自动验证命令，能将破坏性风险降至最低
- **深层逻辑漏洞的系统化调试**：对于那些由于多模块异步交互或边缘状态引起的疑难杂症，Superpowers 的“四阶段根因追踪”技能能引导智能体通过建立假设、编写失败测试、逐步隔离变量的方式找到问题的核心，而非盲目地尝试各种可能的修复方案
- **多智能体并行协作项目（Sub Agent）**：利用 Git Worktrees，多个智能体实例可以同时在不同的物理工作目录中处理同一个仓库的不同任务，互不干扰，显著缩短了大型功能的开发周期

## Superpower 工作流的深度解构

Superpowers 的强大源于其将资深工程经验转化为一套可执行的、结构化的工作流。这个流程从需求萌芽阶段开始，一直持续到代码最终合并进主分支 1。

### **3.1 阶段一：头脑风暴（Brainstorming）与 Socratic 提问**

在任何代码库发生变化之前，brainstorming 技能会被自动触发。智能体会采取一种类似资深架构师的态度，通过提问来挑战用户的原始设想。

| 提问维度               | 目的与预期产出                                               |
| :--------------------- | :----------------------------------------------------------- |
| **品牌与身份识别**     | 明确 UI 的风格倾向（如：是专业现代还是充满活力），确定色值与视觉规范 |
| **用户基数与设备分布** | 决定是采取移动优先还是桌面优先的响应式策略                   |
| **模式对比与选择**     | 提供多种常见的 UI 或架构模式供人类用户选择，并分析优劣       |
| **边缘案例挖掘**       | 询问限流、超时、权限验证等容易被忽视的工程细节               |

这一阶段的最终产出是一份经过验证的设计文档。为了防止信息过载，智能体会分块展示设计细节，确保人类用户有能力在进入下一阶段前对每一部分进行仔细推敲。

### **3.2 阶段二：环境隔离与 Git 工作树（Worktrees）管理**

一旦设计获得批准，Superpowers 不会直接在当前分支操作，而是会自动创建一个独立的 Git 工作树。

这一步骤的工程意义重大：它强制创建了一个干净的实验空间，并在此空间内运行项目设置脚本，验证测试基准。这种物理层面的隔离，不仅防止了未完成代码对主工作区的污染，更允许人类开发者在不同的终端中运行多个 Claude Code 实例，每个实例专注于设计文档中的一个子模块，从而实现真正意义上的 AI 并发编程。

### **3.3 阶段三：精密计划编写（Writing Plans）**

计划阶段是 Superpowers 的“导航中心”。它将高层的设计文档翻译成一系列技术指令。一份典型的 Superpowers 计划文档（PLAN.md）不仅是任务列表，它更像是一份详尽的操作手册。

* **原子化任务定义**：每个任务被限定在 2-5 分钟内完成，确保了极高的可预测性。  
* **明确的文件路径**：预先确定所有需要修改或新增的文件，减少智能体在执行时的搜索开销。  
* **显式代码示例**：在计划中就包含关键代码片段的草案，作为后续执行的参考标准。  
* **验证命令与期望产出**：每个任务都必须配有具体的 curl 命令、npm test 指令或日志检查项，只有产出符合预期，任务才会被标记为完成。

### **3.4 阶段四：子智能体驱动开发（Subagent-Driven Development）**

在执行阶段，Superpowers 引入了角色分离的思想。主智能体（通常是 Claude Code 会话）担任项目经理和架构师的角色，它会派遣“子智能体”去处理具体的原子任务 。

执行过程遵循严格的评审机制。子智能体完成任务后，主智能体会进行两轮评估：首先是规范合规性评审（验证功能是否齐全），其次是代码质量评审（检查是否符合 Clean Code 标准）。这种多主体协作模式模仿了专业工程团队的 Code Review 流程，极大地提高了交付成果的可靠性。

## **第四部分：TDD 铁律——智能体的纪律化灵魂**

测试驱动开发（TDD）在 Superpowers 中被提升到了信仰的高度。这不仅是因为 TDD 本身能减少 bug，更因为它是约束 AI 行为、防止其产生幻觉的最佳工具。

Superpowers 强制执行经典的 **TDD 三步走**，但在 AI 环境下赋予了更强的约束力：

1. **红（RED）**：智能体必须先**编写一个失败的测试**。关键在于，智能体必须实际运行这个测试，并捕捉到错误输出。这是为了证明测试确实在检测预期的行为，而不是因为代码结构问题或拼写错误而“虚假失败”。  
2. **绿（GREEN）**：智能体被允许编写实现代码。这里的原则是“只**写能够让测试通过的最少代码**”。这有效地防止了 AI 产生过度设计或引入未经验证的副作用代码（遵循 YAGNI 原则）。  
3. **重构（REFACTOR）**：在测试保持通过的前提下，智能体对代码进行**清理和优化**。Superpowers 要求在重构前后都要运行全量测试，确保没有引入回归错误。

为了防止智能体像人类一样寻找跳过 TDD 的借口，官方技能文档中明确列出了各种常见理由并予以驳回。

| 常见的跳过借口           | 现实与后果 (Superpowers 的回应)                      |
| :----------------------- | :--------------------------------------------------- |
| “我稍后再补测试”         | 现实：事后测试无法证明代码的正确性，且通过率虚高     |
| “我已经手动测试过了”     | 现实：手动测试不可重复，缺乏长期保障                 |
| “TDD 会让我变慢”         | 现实：长远来看，TDD 减少了调试时间，速度更快         |
| “删除写好的代码太浪费了” | 现实：这属于沉没成本谬误，保留未经验证的代码是技术债 |

这种近乎严苛的约束确保了即便在紧急情况下，智能体也不会牺牲项目的长期健康度。

## 结语

Superpowers 项目的成功在于它深刻洞察了 AI 辅助编程的本质：AI 不是要取代人类的思考，而是要通过吸收人类沉淀下来的最佳工程实践，成为一个更有纪律、更可靠的协作伙伴。通过强制执行设计、计划、测试和评审的闭环，Superpowers 成功地将 AI 从一个“高效的代码打字机”转型为一名“严谨的工程代理人” 。
